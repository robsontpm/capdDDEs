\doxysubsection{Detailed Description}
\subsubsection*{template$<$typename Scalar\+Spec, typename Param\+Spec = Scalar\+Spec$>$\newline
class capd\+::ddes\+::\+Mackey\+Glass$<$ Scalar\+Spec, Param\+Spec $>$}

For now, you need to use class similar to those below to define f(x, y, ..) in your equation. Then you define delays in \textquotesingle{}\mbox{\hyperlink{classcapd_1_1ddes_1_1DiscreteDelaysFunctionalMap}{Discrete\+Delays\+Functional\+Map}}\textquotesingle{} or some other capd\+::ddes\+::\+Functional\+Map derived classes. Please remember that usually \textquotesingle{}\mbox{\hyperlink{classcapd_1_1ddes_1_1DiscreteDelaysFunctionalMap}{Discrete\+Delays\+Functional\+Map}}\textquotesingle{} must specify the Solution template parameter to be able to determine how to handle solution curves.

The setup is much complicated than in the case of O\+D\+Es, as the equations must take into account the problem with the grid over which the solution is defined (as of now). See the problem with the continuity of solution at grid points and long enough integration time described in the papers. Because of those problems, the grid, the solution and the equation (\mbox{\hyperlink{classcapd_1_1ddes_1_1DiscreteDelaysFunctionalMap}{Discrete\+Delays\+Functional\+Map}}) must be aware to some extent of each other. T\+O\+DO\+: (F\+AR F\+U\+T\+U\+RE) try to reduce dependency only to Grid (i.\+e. \mbox{\hyperlink{classcapd_1_1ddes_1_1DiscreteDelaysFunctionalMap}{Discrete\+Delays\+Functional\+Map}} and Solution depend on the Grid, T\+O\+DO\+: (F\+AR F\+U\+T\+U\+RE) but not on each other). It could be done, if \mbox{\hyperlink{classcapd_1_1ddes_1_1DiscreteDelaysFunctionalMap}{Discrete\+Delays\+Functional\+Map}} accepts all solutions that can T\+O\+DO\+: (F\+AR F\+U\+T\+U\+RE) produce concrete Interface of a Jet at Grid\+::\+Time\+Points)

Example\+: typedef Basic\+Doubleton$<$\+I\+Matrix$>$ Set\+Type; typedef D\+D\+E\+Solution\+Curve$<$\+Set\+Type$>$ Solution\+Type; typedef Mackey\+Glass$<$typename Solution\+Type\+::\+Scalar\+Type$>$ F; typedef Discrete\+Delays\+Functional\+Map$<$\+F$>$ R\+HS; typedef typename Solution\+Type\+::\+Grid\+Type Grid\+Type; int p = 128; Grid\+Type grid(2.\+0 / p); F f(2.\+0, 1.\+0, 8.\+0); R\+HS rhs(f, grid(2 $\ast$ p));

We require that solution class has a templated operator of this signature the program will pass in x the m values where m = \mbox{\hyperlink{classcapd_1_1ddes_1_1MackeyGlass_ab128cd5808fc2bddd678d49e56aed8c3}{dimension()}} in fx there will be reference to already initialized vector of dimension d = \mbox{\hyperlink{classcapd_1_1ddes_1_1MackeyGlass_add3d422599f6ed6918e91c7026d83b29}{image\+Dimension()}}. The dimensions are as follows\+: if d = \mbox{\hyperlink{classcapd_1_1ddes_1_1MackeyGlass_add3d422599f6ed6918e91c7026d83b29}{image\+Dimension()}} then usually m = d $\ast$ (number of delayed terms + 1). +1 is for the current term, which is always present, and always assumed to be stored in x\mbox{[}0\mbox{]}. (so if your equation is not dependent on this, then you should not use it in your formulas). In this example (Mackey-\/\+Glass Eq.), we have\+: d = 1 (scalar), m = 2 (two terms\+: current value at 0 and delayed term at t = t -\/ \textbackslash{}tau)

Note that tau is not explicitely present in the equation. The concrete value of the delay is defined when constructing \mbox{\hyperlink{classcapd_1_1ddes_1_1DiscreteDelaysFunctionalMap}{Discrete\+Delays\+Functional\+Map}} from this template function. See docs there.
